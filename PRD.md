## 1. 引言

### 1.1 编写目的

本文档定义「AI 文档校对系统」的软件需求规格，作为研发团队理解系统功能边界、技术约束和验收标准的基准文档。

### 1.2 项目背景

内部团队日常处理大量中文文档（公文、通讯稿、技术文档等），人工校对效率低且易遗漏。本系统通过接入大语言模型（LLM）实现自动化文档校对，以流式方式实时返回校对结果，提升校对效率与质量。

## 2. 系统概述

### 2.1 系统定位

面向内部用户的中文文档自动校对工具，通过 LLM 对文档进行语法、用词、逻辑等维度的校对，实时流式返回分析过程与校对结果。

### 2.2 系统边界

```
┌─────────────────────────────────────────────────────┐
│                    用户（浏览器）                      │
│                                                     │
│   输入文本 / 上传文件  ──→  点击「开始校对」           │
│                                                     │
│   ← 实时展示：思考过程 / 差异对比 / 修改记录           │
└─────────────────────┬───────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│              AI 文档校对系统（本系统）                  │
│                                                     │
│   前端（Vue.js）  ←──SSE──→  后端（Spring Boot）      │
│                                                     │
│   文件解析（PDF/DOCX/TXT）                            │
│   输入清洗与安全过滤                                   │
│   限流与认证                                         │
└─────────────────────┬───────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│            LLM API（DeepSeek / OpenAI 兼容）          │
└─────────────────────────────────────────────────────┘
```

### 2.3 用户角色

| 角色 | 说明 | 权限 |
|------|------|------|
| 匿名用户 | 未配置 API Key 时的默认角色 | 可使用全部校对功能 |
| 认证用户 | 配置了 API Key 后需携带密钥访问 | 可使用全部校对功能 |

### 2.4 核心业务流程

```
用户输入文本或上传文件
        │
        ▼
  前端输入校验（空值 / 长度 / 格式）
        │
        ▼
  文件上传？ ──是──→ 后端解析文件提取文本（PDF/DOCX/TXT）
        │                    │
        否                   │
        │                    │
        └────────┬───────────┘
                 ▼
        发起 POST /api/proofread（SSE）
                 │
                 ▼
        后端安全过滤（认证 → 限流 → 输入清洗 → 长度校验）
                 │
                 ▼
        调用 LLM API（流式 SSE）
                 │
                 ▼
        增量解析三阶段标签（thinking → correction → changes）
                 │
                 ▼
        通过 SseEmitter 实时推送事件到前端
                 │
                 ▼
        前端分阶段渲染：思考面板 → 差异对比 → 修改记录
```

### 3.3 FR-02 文件上传

**描述：** 用户可通过拖拽或点击选择的方式上传文档文件，系统自动解析并提取纯文本。

| 编号 | 验收条件 |
|------|----------|

| FR-02-02 | 支持点击上传区域弹出文件选择对话框 |
| FR-02-03 | 支持的文件格式：`.txt`、`.docx`、`.pdf` |
| FR-02-04 | 单文件大小上限 20MB |
| FR-02-05 | 文件格式不正确时提示"不支持的文件格式" |
| FR-02-06 | 文件过大时提示文件大小和上限 |
| FR-02-07 | 上传后显示文件名和文件大小 |
| FR-02-08 | 已选择文件后显示文件信息卡片，可点击关闭按钮清除 |
| FR-02-09 | 文件解析成功后，提取的文本自动填入文本输入框 |
| FR-02-10 | 文件内容为空时提示"文件内容为空" |
| FR-02-11 | 拖拽悬浮时上传区域显示蓝色高亮反馈 |
| FR-02-12 | 文件解析进行中时，上传组件显示禁用状态 |

**文件解析规则：**

| 格式 | 解析方式 | 安全校验 |
|------|----------|----------|
| `.txt` | 直接以 UTF-8 编码读取 | Magic Byte 校验（排除 ELF/PE/Java class 等二进制格式）、前 8KB 无 null 字节 |
| `.docx` | Apache POI 提取段落文本 | ZIP Magic Byte 校验（PK 签名） |
| `.pdf` | Apache PDFBox 提取文本 | PDF Magic Byte 校验（%PDF 签名） |

**上传区域界面要素：**
- 虚线边框圆角矩形
- 云上传图标 + 提示文字"拖拽文件到此处，或点击选择"
- 副标题显示支持的格式和大小限制

### 3.4 FR-03 AI 校对

**描述：** 用户点击「开始校对」按钮后，系统调用 LLM API 对文本进行校对。

| 编号 | 验收条件 |
|------|----------|
| FR-03-01 | 文本为空或纯空白时，按钮置灰不可点击 |
| FR-03-02 | 文本超过 10000 字符时，按钮置灰不可点击 |
| FR-03-03 | 校对进行中时，按钮显示加载状态（蓝色 + "AI 正在校对中..."） |
| FR-03-04 | 校对进行中时，可发起新的校对请求（自动取消前一个） |
| FR-03-05 | 校对完成后，按钮恢复为可点击状态 |

**LLM 调用参数：**

| 参数 | 值 | 说明 |
|------|------|------|
| temperature | 0.3 | 低随机性，确保校对结果稳定 |
| stream | true | 启用流式输出 |
| system prompt | 结构化校对指令 | 要求三阶段 XML 标签输出格式 |

**System Prompt 输出格式要求：**

LLM 必须按以下顺序输出三个独立的顶层 XML 标签：

1. `<thinking>...</thinking>` — 文档分析与校验规则推演
2. `<correction>...</correction>` — 校对后的完整文档
3. `<changes>...</changes>` — 修改记录 JSON 数组

`<thinking>` 内容必须包含四个结构化板块：
- 文档体裁分析
- 目标受众
- 正式程度
- 校验规则（3-5 条）

`<changes>` 格式：
```json
[
  {
    "original": "原词句",
    "corrected": "修改后",
    "reason": "修改理由"
  }
]
```

无修改时 `<changes>` 输出空数组 `[]`。

### 3.8 FR-07 流式传输

**描述：** 全链路采用 SSE（Server-Sent Events）实现流式传输，用户可实时看到校对进度。

| 编号 | 验收条件 |
|------|----------|
| FR-07-01 | 前端通过 `fetch` API 发起 POST 请求，以 SSE 方式接收响应流 |
| FR-07-02 | SSE 直连后端（绕过开发代理），避免代理缓冲导致流式失效 |
| FR-07-03 | 流式事件分为 7 种类型：start / thinking / correction / change / changes / done / error |
| FR-07-04 | thinking 和 correction 事件支持增量更新（同一事件多次发送，逐步累积内容） |
| FR-07-05 | change 事件逐条发送单个修改项（增量模式） |
| FR-07-06 | changes 事件在流结束时发送完整修改列表 |
| FR-07-07 | 流式传输过程中，前端正确处理标签截断、不完整 JSON 等边界情况 |
| FR-07-08 | SSE 连接支持 `AbortController` 取消 |

**SSE 事件协议：**

| 事件名 | 数据格式 | 触发时机 | 可重复 |
|--------|----------|----------|--------|
| `start` | `{}` | 流开始 | 否 |
| `thinking` | `{text: string, done: boolean}` | 分析过程 | 是（增量） |
| `correction` | `{text: string, done: boolean}` | 校对结果 | 是（增量） |
| `change` | `{change: Change, done: boolean}` | 单条修改 | 是（增量） |
| `changes` | `{changes: Change[], done: boolean}` | 完整修改列表 | 否 |
| `done` | `{}` | 流结束 | 否 |
| `error` | `{error: string}` | 错误 | 否 |

**Change 数据结构：**
```typescript
interface Change {
  original: string   // 原文词句
  corrected: string  // 修改后词句
  reason: string     // 修改理由
}
```

| 服务端未知错误 | "服务器内部错误，请稍后重试" |

## 5. 接口需求

### 5.1 前端内部接口

#### 5.1.1 API 客户端模块 (`api/index.js`)

event:change

data:{}

## 7. 安全需求

### 7.1 认证与授权

| 编号 | 要求 |
|------|------|
| SEC-01 | 支持 Bearer Token 和 X-API-Key Header 两种认证方式 |
| SEC-02 | 未配置 API Key 时自动跳过认证（仅限开发环境） |
| SEC-03 | 认证失败返回 401 状态码和 WWW-Authenticate 响应头 |

### 7.2 输入安全

| 编号 | 要求 |
|------|------|
| SEC-10 | 用户输入在发送给 LLM 前必须经过 `InputSanitizer` 清洗 |
| SEC-11 | 清洗流程：剥离危险 XML 标签 → 检测注入模式 → 角括号全角编码 |
| SEC-12 | 注入模式检测必须覆盖中英文双语（14 种以上模式） |
| SEC-13 | 用户输入必须用 XML 标签（`<user_document>`）包裹，与系统指令隔离 |
| SEC-14 | 文件上传必须进行 Magic Byte 校验，防止伪造扩展名 |
| SEC-15 | 后端强制校验文本长度上限（10000 字符），不依赖前端校验 |

### 7.3 限流与防刷

| 编号 | 要求 |
|------|------|
| SEC-20 | 校对接口限流：每 IP 每分钟最多 10 次请求 |
| SEC-21 | 文件解析接口限流：每 IP 每分钟最多 20 次请求 |
| SEC-22 | 超过限流返回 429 状态码和 Retry-After 响应头 |
| SEC-23 | 默认使用 `remoteAddr` 作为客户端标识（防 IP 欺骗），可配置为信任代理头 |
| SEC-24 | 限流存储上限 100,000 条目，超出自动淘汰过期条目 |
| SEC-25 | 并发活跃 SSE 连接数上限 30 |
| SEC-26 | 限流条目字段使用 `volatile` 修饰，确保并发可见性 |

### 7.4 信息脱敏

| 编号 | 要求 |
|------|------|
| SEC-30 | 所有返回给前端的错误信息必须为友好模糊提示，禁止包含：异常类名、堆栈信息、内部主机名/IP、API Key 片段、库版本信息 |
| SEC-31 | LLM API 原始错误体仅记录到服务端日志，不透传前端 |
| SEC-32 | 全局异常处理器（GlobalExceptionHandler）返回通用错误消息 |
| SEC-33 | SSE 错误事件使用 ObjectMapper 安全序列化，禁止手动字符串拼接 JSON |

### 7.5 超时与资源保护

| 编号 | 要求 |
|------|------|
| SEC-40 | SSE 连接超时 5 分钟，超时后必须取消底层 OkHttp 调用 |
| SEC-41 | OkHttp 调用总超时（callTimeout）6 分钟 |
| SEC-42 | OkHttp 连接超时 30 秒，读取超时 5 分钟，写入超时 30 秒 |
| SEC-43 | 文件解析超时 30 秒 |
| SEC-44 | 线程池有界（禁止 CachedThreadPool），防止线程耗尽 |
| SEC-45 | Tomcat 连接池加固：max-threads=50, max-connections=200, connection-timeout=10s |
| SEC-46 | 客户端断开连接时（SSE onError），必须取消 OkHttp 调用并递减连接计数 |
| SEC-47 | Error Emitter 也必须设置超时（5 分钟），防止无限挂起 |

### 7.6 Web 安全头

| 编号 | Header | 值 |
|------|--------|-----|
| SEC-50 | X-Frame-Options | DENY |
| SEC-51 | X-Content-Type-Options | nosniff |

| SEC-54 | Strict-Transport-Security | max-age=31536000; includeSubDomains |
| SEC-55 | Referrer-Policy | strict-origin-when-cross-origin |
| SEC-56 | Permissions-Policy | camera=(), microphone=(), geolocation=() |

| SEC-60 | 允许来源通过环境变量 `CORS_ORIGINS` 配置，默认为本地开发地址 |
| SEC-61 | 允许的请求头限制为：Content-Type、X-API-Key、Authorization、Accept |

## 9. 约束与假设

### 9.1 约束

| 编号 | 约束 |
|------|------|
| CON-01 | LLM API 为外部服务，其响应时间和可用性不受本系统控制 |
| CON-02 | 单次校对文本上限 10000 字符，由 LLM 上下文窗口和响应时间决定 |
| CON-03 | 系统无持久化存储，不保存历史校对记录 |
| CON-04 | 系统为单实例部署，不支持水平扩展（限流/连接计数基于内存） |
| CON-05 | 文件解析在内存中完成，20MB 文件约占用 60-80MB 堆内存 |

### 9.2 假设

| 编号 | 假设 |
|------|------|
| ASM-01 | LLM API 兼容 OpenAI Chat Completions 接口格式 |
| ASM-02 | LLM 能够遵循 System Prompt 中的三阶段 XML 标签输出格式 |
| ASM-03 | 用户上传的文件内容为合法的 UTF-8 编码文本 |
| ASM-04 | 部署环境的网络可访问 LLM API 端点 |
